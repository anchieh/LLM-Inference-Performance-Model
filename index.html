理論依據 v5.0

核心公式

TPS = BW × util × sizeF / traffic
traffic = active_weights + KV(L)

Decode 對所有裝置都是記憶體頻寬瓶頸：每產生一個 token，全部活躍權重都要從記憶體讀一次。CPU / iGPU / NPU 打的是同一條 DRAM 匯流排，差異只在利用率 util——這是 v5 對 v4 reuse 體系的修正：共享記憶體平台上 NPU 不會因「重複利用」而超越頻寬上限。

算力三層

頻寬上限（物理）→ 利用率（工程）→ runtime（生態）

理解算力的心智模型：第一層是記憶體頻寬給出的物理天花板，任何裝置都翻不過去；第二層是硬體工程決定能吃到天花板的幾成（util）；第三層是軟體生態決定第二層能否兌現（同一顆 Jetson，runtime 不同差 2–3 倍）。比較平台時先比第一層，再看第二、三層有沒有拖後腿——TOPS 行銷數字不在這三層裡。

util 校準係數

util 是半經驗校準常數，依平台家族與裝置各自標定，範圍 0.31–0.87。新世代平台的 iGPU 與 NPU 利用率顯著高於前代；U 系列因功耗限制較 H 系列低約三成。標「校準」代表該家族有實測基礎；標「外推」代表僅由規格推算。

首字延遲 FTL

FTL = coeff × P × (L / 1024)

Prefill 是計算瓶頸，與 decode 完全不同。coeff（ms/B，1k 輸入基準）依裝置與世代標定：新世代 iGPU 約 45–65，前代約 290–630；NPU 在新世代約 190，前代約 650——NPU 的 prefill 到最新世代才真正可用。CPU 沿用 2PL/FLOPS。

NPU Runtime 成熟度

硬體能力 ≠ 可得性能。NPU 的 LLM 支援依 runtime 生態（驅動、編譯器、框架）差異極大：標「成熟」代表該世代 NPU 已可穩定接近 iGPU 水準；標「實驗性」代表 decode 約為 iGPU 的五到七成、prefill 明顯落後，採購決策不應以 NPU 數字為準。

dGPU 模型

TPS = VRAM_BW × 0.60–0.70 / traffic
if model + KV > VRAM → warning

dGPU decode 由 VRAM 頻寬主導。模型＋KV 塞不進 VRAM 時進入 offload 情境——該區誤差可達 2–3 倍，本工具只給警告不給數字。

KV Cache（GQA-aware）

KV = kv_per_token × L

kv_per_token 由模型架構（層數 × KV heads × head_dim）決定，模型 preset 自動帶入；不同架構同參數量可差十倍。KV 計入 RAM/VRAM 需求，decode 時 KV 讀取流量併入 model_bytes。

✗ 不適用範圍

多節點分散式推論。VRAM 不足的 offload 吞吐數字。跨陣營 NPU 對比。M.2/PCIe AI 加速卡（無可信公開 LLM 數據、runtime 生態未成熟，故不提供估算）。Batch serving 吞吐最佳化。投機解碼等 runtime 加速技術。

使用限制與但書
本工具為 semi-empirical 工程估算器：以公開規格＋校準常數推算，非實測保證。「校準」精度目標 ±15%，「外推」±25%。實際效能受 runtime 版本、驅動、散熱設計、記憶體時序影響。採購決策請以實機驗證為最終依據。
